# Handoff — 2026-04-17 — Single Parent Pipeline Cleanup

This note records the cleanup and consolidation work after the parent FreeRouting path was found to diverge from the main parent composition pipeline.

---

## 1. What was completed

### Parent pipeline consolidation
The hierarchical parent flow now uses a single parent composition / stamping / routing path.

Previously:
- `run_hierarchical_pipeline.py` used `compose_subcircuits.py` for a composition snapshot
- then separately invoked `demo_hierarchical_freerouting.py` for the visible parent routing stage

That duplication allowed the visible parent path to drift and produce invalid DSN inputs where child modules could be outside the stamped board outline.

Now:
- `run_hierarchical_pipeline.py` invokes `compose_subcircuits.py` directly for the parent routing stage
- the parent routing stage uses:
  - packed composition
  - parent geometry validation
  - parent stamping
  - parent routing
  - parent artifact persistence
  - parent preview rendering

This means the hierarchical visible run and the parent artifact pipeline now share the same implementation path.

### Explicit parent geometry validation
`compose_subcircuits.py` now validates parent geometry before stamping/routing.

Validation checks include:
- non-degenerate board outline
- component bodies inside outline
- pads inside outline
- traces inside outline
- vias inside outline

If validation fails:
- the run fails explicitly
- structured validation details are emitted
- the pipeline does not silently continue to DSN export / FreeRouting

This directly addresses the earlier failure mode where FreeRouting loaded a board with many modules outside the PCB.

### Parent preview rendering moved into the single pipeline
The unified parent path now renders parent previews itself.

Current outputs under the parent artifact render directory include:
- `parent_stamped.png`
- `parent_routed.png`

These are generated by the same path that stamps and routes the parent board.

### Obsolete demo path hard-disabled
`demo_hierarchical_freerouting.py` was replaced with a hard-failure shim.

It now:
- explains that the script is obsolete
- points callers to:
  - `compose_subcircuits.py`
  - `run_hierarchical_pipeline.py`
- exits nonzero

This prevents any lingering stale caller from silently using the old alternate parent pipeline.

### GUI / analysis cleanup toward unified artifacts
References were updated to prefer unified parent artifacts and metadata:
- `gui/pages/analysis.py`
- `gui/components/progression_viewer.py`
- `autoexperiment.py`

The UI copy now refers to:
- stamped parent
- routed parent
- unified parent pipeline

instead of the older demo/preloaded/freerouted framing where appropriate.

## 2. Root cause that was fixed

The user observed a FreeRouting view where most parent child modules were outside the PCB.

Root cause:
- the visible parent routing stage had diverged from the main parent composition pipeline
- it moved footprints into parent-composition coordinates
- but preserved an incompatible base-board `Edge.Cuts` outline
- so FreeRouting loaded a DSN whose board outline did not actually contain the placed child modules

This was a real pipeline bug, not just a misleading render.

The fix was to eliminate the alternate visible parent routing implementation from the hierarchical runner and route everything through the single parent pipeline.

`demo_hierarchical_freerouting.py` still exists in the repo, but only as a hard-failure placeholder.

This is intentional for now so stale callers fail loudly.

Possible later cleanup:

There are still fields/variables like:
- `visible_output_dir`
- `skip_visible`

in places such as `autoexperiment.py` that now conceptually refer to the unified parent pipeline rather than a separate visible/demo stage.

These are not functionally wrong, but they are semantically stale and should be renamed in a cleanup pass.

### C. Autoexperiment still has some legacy path assumptions
`autoexperiment.py` was partially updated to use unified parent outputs, but it likely still deserves a focused cleanup pass to:
- rename variables
- stop assuming old nested `visible_parent/visible_parent` layouts
- prefer parent artifact metadata/debug payloads directly

### D. GUI still supports some backward-compatible preview fallbacks
The GUI preview discovery still accepts older filenames for compatibility.

That is useful for existing artifacts, but eventually the code can be simplified once old artifact formats are no longer needed.

### E. Parent routing is still often rejected
The unified parent pipeline is now structurally correct, but the routed parent can still be rejected for real reasons such as:
- `missing_required_anchors`
- `illegal_routed_geometry`

That is now a real engineering problem to improve, rather than a pipeline-path inconsistency problem.
