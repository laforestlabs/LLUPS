## 2. The key problem still remaining

The monitor is better, but it is still not inspection-grade or fully trustworthy during long runs.

### Problem A — preview selection still relies too much on filesystem guessing
The current monitor still uses fallback preview discovery logic that guesses from known paths.

That means:
- the monitor can show stale images
- the monitor can show the wrong image for the current stage
- the monitor can lag behind what the pipeline is actually doing
- some paths still assume older artifact layouts

### Problem B — parent preview discovery still contains stale assumptions
There are still places where preview discovery depends on older conventions and, in at least one place, a hardcoded parent artifact slug was used during the first pass.

That is not acceptable long-term.

The monitor should not depend on:
- a fixed parent artifact slug
- legacy nested visible-parent directories
- old demo-era naming assumptions

### Problem C — status text is richer, but still not “live operator console” quality
The user wants to see what the system is doing in realtime.

The current status is improved, but it still needs to feel more like:
- a live operator console
- a current-task dashboard
- a visual progress tracker

Specifically, the monitor should make it obvious when the system is:

- solving a leaf
- waiting on leaf routing
- composing the parent
- stamping the parent
- routing the parent
- validating the parent
- scoring the round
- idle / done / failed

### Problem D — event stream is still too generic
The event panel currently mostly reflects round-level JSONL activity.

It should become more useful for long-running runs by showing stage transitions and current work in a more explicit way.

### Problem E — there is still no strong “current image source of truth”
The monitor should prefer the live `preview_paths` emitted by status over filesystem heuristics.

That is the cleanest architecture:
- pipeline emits current preview paths
- monitor renders those exact paths
- fallback discovery is only for backward compatibility

## 3. Recommended next implementation step

### Priority 1 — make `preview_paths` the primary source of truth in the monitor
This is the most important next step.

The monitor should:

1. read `preview_paths` from live status
2. use those paths first for display
3. only fall back to filesystem discovery if `preview_paths` is missing

Recommended preview keys to support explicitly:

- `leaf_preview`
- `parent_stamped_preview`
- `parent_routed_preview`
- `leaf_artifact_dir`
- `parent_artifact_dir`

Acceptance criteria:
- the monitor shows the image that corresponds to the current stage
- no hardcoded parent artifact slug is needed
- no stale demo-path assumptions are needed for current runs

Acceptance criteria:

Goal:
- make the event panel useful during long runs

Recommended behavior:
- append explicit stage-transition events when status milestones change
- show timestamps
- show round number
- show current leaf / parent when available
- show success/failure markers for each stage

Examples of useful event titles:
- `Round 3 started`
- `Leaf solve launched`
- `Leaf solve complete`
- `Parent composition launched`
- `Parent routing launched`
- `Parent routing complete`
- `Round 3 scored`
- `Round 3 kept`
- `Round 3 discarded`
- `Run complete`

Acceptance criteria:
- the event panel visibly advances during long runs
- the user can see the last few meaningful actions without opening logs

### Priority 4 — improve live image layout and labeling
Goal:
- make the live image area more informative

Recommended layout:
- left: live leaf preview
- right: live parent preview

Recommended labels:
- `Live leaf / current best preview`
- `Live stamped parent preview`
- `Live routed parent preview`

Recommended metadata under each image:
- filename
- artifact dir
- stage relevance
- whether it is current or fallback

Acceptance criteria:
- the user can tell what each image represents
- the user can tell whether the parent image is stamped or routed
- the user can tell whether the image is current-stage or fallback

### Priority 5 — add explicit “no new preview yet” states
Goal:
- avoid the impression of a hang when the pipeline is still working but no new image exists yet

Recommended behavior:
- if the current stage has no preview yet, show:
  - a text explanation
  - the current action
  - the current command
  - the last available preview if useful
- distinguish:
  - `no preview yet for this stage`
  - `using previous preview while current stage runs`
  - `preview unavailable due to failure`

Acceptance criteria:
- the user no longer mistakes “no new image yet” for “the app is hung”
